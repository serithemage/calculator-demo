# 프로젝트 개발 원칙 및 규칙 (Project Rules)

이 문서는 **Scientific Calculator** 프로젝트의 코드 품질과 일관성을 유지하기 위한 핵심 원칙을 정의합니다. 모든 개발자와 AI 어시스턴트는 이 규칙을 준수해야 합니다.

## 1. 핵심 개발 원칙

### 1.1 TDD (Test-Driven Development) 필수 적용
- **대상**: UI 컴포넌트를 제외한 모든 **코어 로직** (계산 엔진, 수식 파서, 숫자 포매터, 입력 검증기 등).
- **작업 사이클**:
    1. **Red**: 구현할 기능에 대한 실패하는 테스트 코드를 먼저 작성한다.
    2. **Green**: 테스트를 통과할 수 있는 최소한의 코드를 작성한다.
    3. **Refactor**: 코드의 기능을 유지하면서 구조를 개선하고 중복을 제거한다.
- **목표**: 코어 로직에 대해 100% 테스트 커버리지를 유지한다.

### 1.2 SOLID 원칙 준수
객체 지향 설계 및 함수형 프로그래밍 시 다음 원칙을 엄격히 따릅니다.
- **S (Single Responsibility)**: 각 클래스/함수는 단 하나의 명확한 책임만 가진다.
- **O (Open/Closed)**: 기존 코드를 수정하지 않고도 새로운 연산자나 함수를 확장할 수 있도록 설계한다.
- **L (Liskov Substitution)**: 상속이나 인터페이스 구현 시 하위 타입이 상위 타입을 오류 없이 대체할 수 있어야 한다.
- **I (Interface Segregation)**: 범용적인 하나의 인터페이스보다 구체적인 여러 개의 인터페이스를 지향한다.
- **D (Dependency Inversion)**: 구체적인 구현이 아닌 추상화(인터페이스)에 의존한다. 의존성 주입(DI)을 적극 활용한다.

## 2. 기술 스택 및 표준

- **Framework**: React 18 (TypeScript 환경)
- **Styling**: Tailwind CSS (Tech Spec에 정의된 디자인 시스템 준수)
- **Math Engine**: Math.js (표준 수학 연산 및 상수 처리)
- **Testing**: Vitest (코어 로직 단위 테스트 전용). **UI/E2E 자동화 테스트 도구 제외**.
- **Deployment**: GitHub Actions를 통한 GitHub Pages 배포

## 3. 워크플로우 (Workflow)

- 모든 코어 기능 구현은 테스트 코드가 먼저 존재해야 한다.
- **UI 구현 시에는 자동화 테스트를 작성하지 않고**, 브라우저에서 직접 동작을 확인한다.
- UI 통합 전 반드시 **코어 로직 테스트**가 통과(`Green`) 상태임을 확인한다.
- PRD나 Tech Spec에서 변경이 생길 경우, 관련 문서를 먼저 업데이트한 후 개발에 착수한다.
